# DEVELOPMENT_RULES.md

Coding standards, conventions, and workflow rules for this codebase. Mostly Level 1 (Claude's call, per [MASTER_BLUEPRINT.md](MASTER_BLUEPRINT.md) §8) — documented so conventions stay consistent across a long project, not because each one needs sign-off.

Status: **CSS/JS approach, Form Request pattern, security, database, authorization (Policy/middleware), file-upload, service-layer, repository-layer, DTO, caching, row-locking, business-rule-validation, audit-logging, soft-delete, polymorphic-relation, and config-overlay conventions confirmed in practice through Steps 1–11.**
Last updated: 2026-07-03

## Code Style

- **PSR-12**, enforced via **Laravel Pint** (already installed as a dev dependency — `./vendor/bin/pint` before considering a piece of work done).
- 4-space indentation, no tabs (PHP/Blade default).
- Strict typing where practical (`declare(strict_types=1)` in new PHP classes — Level 1, standard modern-Laravel practice).

## Naming Conventions

| Thing | Convention | Example |
|---|---|---|
| Models | Singular, PascalCase | `Product`, `VendorProfile` |
| Tables | Plural, snake_case | `products`, `vendor_profiles` |
| Controllers | PascalCase + `Controller` suffix | `ProductController` |
| Routes (named) | dot notation, role-scoped where relevant | `vendor.products.index` |
| Blade views | kebab-case, mirrors route structure | `resources/views/auth/login.blade.php` |
| Blade components | PascalCase class / kebab-case tag | `<x-role-card>` → `RoleCard.php` |
| DB columns | snake_case | `created_at`, `approval_status` |
| JS/CSS variables | camelCase / kebab-case respectively | `isOpen`, `--space-3` |

## Folder Structure Conventions

- Controllers grouped by role where it clarifies intent once those areas exist: `app/Http/Controllers/Vendor/` (built Step 7: `DashboardController`, `StoreController`, `ProductController` — deliberately kept separate, not combined into one "vendor" controller), `app/Http/Controllers/Admin/` (built Step 11: `DashboardController`, `UserController`, `VendorController`, `ProductController`, `CategoryController`, `ReportController`, `RecommendationAnalyticsController`, `AuditLogController`, `SettingsController`, `HealthController` — one controller per admin responsibility area, each thin, delegating to a matching `Admin\*Service`). Guest/buyer controllers stay unprefixed at `app/Http/Controllers/` since the buyer area is the main site. **Form Requests follow the same role subfolder convention** — `app/Http/Requests/Vendor/` (`StoreProductRequest`, `UpdateProductRequest`, `UpdateStoreRequest`), `app/Http/Requests/Admin/` (`StoreCategoryRequest`, `UpdateCategoryRequest`, `VendorModerationRequest`, `ProductModerationRequest`, `UpdateSettingsRequest`), matching the controllers they validate for. **Naming-collision note:** `App\Http\Controllers\Admin\*` classes share short names with existing unprefixed/`Vendor\*` controllers (`CategoryController`, `ProductController`, `VendorController`, `DashboardController`) — every route file import that needs both aliases one of them (`use App\Http\Controllers\Admin\CategoryController as AdminCategoryController;`), same pattern already established for `Vendor\DashboardController` vs. a hypothetical `Admin\DashboardController` before Step 11 made it a real collision.
- Validation via **Form Request** classes (`app/Http/Requests/`), not inline `$request->validate()` in controllers, once forms have more than a couple of trivial rules.
- Authorization via **Policies** (`app/Policies/`) once role-gated *or ownership-gated* actions exist. **First one built 2026-07-02 (Step 6):** `AddressPolicy` (`update`/`delete`, checking `$user->id === $address->user_id`), used via `$this->authorize()` in `AddressController`. This isn't just role-gating (vendor vs. buyer) — plain "a user can only touch their own resource" ownership checks are exactly what Policies are for too.
  **Bug fixed in the process:** Laravel 11+'s slim base `Controller` class no longer includes the `AuthorizesRequests` trait by default (unlike older Laravel/Breeze scaffolding), so `$this->authorize()` didn't exist until it was added back to `app/Http/Controllers/Controller.php`. Any future controller needing `$this->authorize()` now has it — don't re-diagnose this if it comes up again.
  **Extended 2026-07-03 (Step 7):** `ProductPolicy` and `StorePolicy` follow the same ownership-check pattern. **Note the naming exception:** `StorePolicy` protects `VendorProfile`, not a `Store` model — Laravel's policy auto-discovery only works when the policy is named `{Model}Policy` (`VendorProfilePolicy` would auto-resolve; `StorePolicy` does not). Registered explicitly via `Gate::policy(VendorProfile::class, StorePolicy::class)` in `AppServiceProvider::boot()`. If a policy class name doesn't match its model, it needs this same manual registration — don't assume auto-discovery covers it.
  **Extended 2026-07-03 (Step 10):** `CartPolicy` (protects `CartItem`, same naming-exception registration as `StorePolicy` above) and `OrderPolicy` (protects `Order` — auto-discovers correctly, `Order` → `OrderPolicy` matches convention — **and** `OrderItem`, which needs the same manual registration since `OrderItem` → `OrderPolicy` does not match). **New pattern here:** one Policy class can legitimately answer authorization questions for two closely-related models (`Order` and `OrderItem`) when both questions boil down to the same underlying check ("does this user have a legitimate reason to touch this order or its line items") — register the same policy class against both models via `Gate::policy()` rather than duplicating near-identical logic into a second class.
- **Role-gating middleware, added Step 7, extended Step 11:** `EnsureUserIsVendor` (alias `vendor`) restricts an entire route group to a specific `users.role` value — distinct from Policies, which gate access to a *specific resource instance*. Use middleware for "can this role see this section of the site at all" and Policies for "can this user touch this specific record." **`EnsureUserIsAdmin` (alias `admin`, Step 11)** follows the identical one-file `abort_unless()` shape, but checks **two** conditions, not one — `$user->role === 'admin'` **and** `$user->status === 'active'` — so a suspended administrator is locked out exactly like a suspended buyer/vendor. No admin-area Policy exists: the whole `/admin/*` group is middleware-gated only, since administrators supervise every record rather than a subset they own (there's no ownership question a Policy would need to answer).
- **Admin service namespace, introduced Step 11:** `app/Services/Admin/` groups every admin-only service (`CategoryService`, `UserManagementService`, `VendorManagementService`, `ProductModerationService`, `ReportService`, `RecommendationAnalyticsService`, `AuditLogService`, `SettingsService`, `HealthCheckService`) plus `app/Services/Admin/Dashboard/` (`DashboardService` + its 6 widgets + a `Concerns/BuildsMonthlySeries` trait) — the same "one sub-namespace per cohesive subsystem" pattern first established for `app/Services/Recommendation/` in Step 9. One service class per admin responsibility area, never a single catch-all `AdminService`.
- **Dashboard widget pattern, introduced Step 11:** `DashboardService` doesn't compute anything itself — it only calls 6 single-purpose widget classes (`UsersWidget`, `VendorWidget`, `OrdersWidget`, `RevenueWidget`, `RecommendationWidget`, `SystemHealthWidget`), each independently responsible for fetching its own data. A shared `BuildsMonthlySeries` trait holds the one piece of genuinely duplicated logic (the last-6-months GROUP-BY-month query) so the 5 widgets that need a monthly chart series don't each reimplement it. Use this widget-per-concern shape for any future dashboard-style page that aggregates data from several unrelated sources.
- **Audit logging, introduced Step 11:** `AuditLogService::record()` is called explicitly from inside every `Admin\*Service` write method — never via an Eloquent Observer. An observer on `User`/`Product`/`Category`/`VendorProfile` would also fire for ordinary buyer/vendor self-service edits (a buyer updating their own name, a vendor editing their own product), which must never appear in the administrator audit trail; an explicit call site is the only way to guarantee that distinction. `AuditLog.auditable` is this project's first polymorphic relation (`nullableMorphs('auditable')` in the migration, `morphTo()` on the model) — reach for a polymorphic relation over one-nullable-FK-per-entity when the list of "things this can point to" is expected to keep growing (a new moderated entity type here shouldn't need a schema migration).
- Blade components in `resources/views/components/`, one file per component, matching the names introduced in [DESIGN_SYSTEM.md](DESIGN_SYSTEM.md) and the Component Trees in [UI_BLUEPRINT.md](UI_BLUEPRINT.md).
- Planning docs live in `docs/` at project root (see [MASTER_BLUEPRINT.md](MASTER_BLUEPRINT.md) §12) — kept out of `app/`, `resources/`, etc. entirely.
- **Added 2026-07-02 (Step 4):** global helper functions live in `app/helpers.php`, autoloaded via `composer.json`'s `autoload.files` (requires `composer dump-autoload` after editing). Used sparingly — `gated_route()` (every gated nav link/CTA calls this to resolve to `/login` for guests, or a real page / `/coming-soon` for authenticated users — extended Step 8 to accept an optional real-destination argument) and `cart_count()` (Step 8: summed the session cart; **rewritten Step 10** to query the buyer's persistent `Cart` instead, now that one exists). Prefer a class/service over a new global helper if the logic grows more complex than a one-line lookup.
- **`<x-flash-status>` renders validation errors too, since 2026-07-03 (Step 10):** originally only `session('status')` (a success/info message). Extended to also loop over Laravel's `$errors` bag (styled as a red variant) after a real gap was found: a service-layer `ValidationException` (see Database Conventions below) correctly blocked an action but the page gave no visible reason why. Since this component already sits on every major page, any future `ValidationException` thrown from a service will automatically surface correctly — no per-page error markup needed.
- **Added 2026-07-02 (Step 4):** authentication controllers live under `app/Http/Controllers/Auth/` (`LoginController`, `RegisterController`), matching the existing Login/Registration Form Requests already in `app/Http/Requests/`. Credential-checking logic (`Auth::attempt`, rate limiting) lives on the Form Request itself (`LoginRequest::authenticate()`), not the controller — standard Laravel/Breeze convention, keeps the controller a thin orchestrator.
- **Service layer, introduced 2026-07-03 (Step 8):** `app/Services/` for query/business logic too complex or too shared-across-controllers to belong in any single controller. First one: `ProductCatalogueService` — every catalogue-related controller (`ProductCatalogueController`, `CategoryController`, `SearchController`, `VendorController`) calls into it rather than building queries directly, specifically so Step 9's recommendation logic has a clean, already-established query foundation to sit alongside rather than duplicate. Rule of thumb: if the same non-trivial query logic would otherwise be copy-pasted across 2+ controllers, it belongs in a Service.
- **Repository layer, introduced 2026-07-03 (Step 9):** `app/Repositories/` for data-access logic that multiple services need but that isn't itself business logic — `RecommendationRepository` is the sole point of contact for the `user_interactions`/`recommendation_logs` tables. Every algorithm service (`ContentBasedService`, `CollaborativeFilteringService`, `PopularityService`) goes through it rather than querying those tables directly, so the storage shape can change (e.g. adding a new interaction type, or eventually reintroducing precomputed aggregate tables) without touching algorithm code. Distinct from a Service: a Repository only reads/writes one storage concern, it doesn't apply scoring/business rules.
- **DTOs, introduced 2026-07-03 (Step 9):** `app/DTOs/` for `final`, `readonly` value objects that cross a layer boundary with a fixed, predictable shape — `RecommendationScore` and `RecommendationResult`. Used when a plain array or an Eloquent model would either leak internal detail (a model would expose every column) or be too loosely typed (an array has no compiler-checked shape) for what's being passed between the algorithm services, the Hybrid blender, and Blade templates. Not a project-wide convention yet — only introduced where a value genuinely needs to be immutable and strongly shaped; don't reach for a DTO where a model or array is already sufficient.
- **Recommendation namespace, introduced 2026-07-03 (Step 9):** `app/Services/Recommendation/` groups every recommendation-specific service (`InteractionTrackingService`, `ContentBasedService`, `CollaborativeFilteringService`, `PopularityService`, `HybridRecommendationService`, `RecommendationService`, `RecommendationCacheService`, `RecommendationEvaluator`) under one sub-namespace rather than flat in `app/Services/` alongside `ProductCatalogueService` — signals that these classes only ever get used together, and keeps the recommendation subsystem's surface area easy to find as one unit. `RecommendationService` is the **only** class controllers/views should depend on directly; the other seven are internal collaborators. Apply the same "one sub-namespace per cohesive subsystem" pattern if a future phase (e.g. Checkout/Payments in Step 11) similarly grows more than 3–4 tightly-related services.
- **Cache versioning, introduced 2026-07-03 (Step 9):** `RecommendationCacheService` uses version-counter cache keys (`recommendations:v{globalVersion}:{key}:user{id}:uv{userVersion}`) rather than `Cache::tags()`, since this project's `CACHE_STORE=database` doesn't support tagging (only Redis/Memcached do). Bumping a version number (stored itself in the cache, `Cache::forever()`) makes every previously-cached key for that scope unreachable without needing to enumerate or delete them — a portable pattern that works on any cache driver. Use this pattern for any future feature needing scoped cache invalidation on a non-tagging driver, rather than assuming `Cache::tags()` is available.
- **Checkbox gotcha, fixed 2026-07-03 (Step 8):** an HTML checkbox that's unchecked is simply absent from the submitted form data — `$request->validated()`/`->boolean()` can't tell "user unchecked this" from "this field isn't part of the form." `<x-checkbox>` now renders a hidden `<input type="hidden" value="0">` immediately before the real checkbox (which explicitly sets `value="1"`) so unchecking always submits a real `"0"`. Apply this pattern to any future raw (non-component) checkbox in the app.
- **Session-based state without a DB table, established 2026-07-03 (Step 8):** the cart (`CartController`) stores `product_id => quantity` directly in the Laravel session rather than a `carts` table, since Step 10 (Shopping Cart) owns that schema decision. This is an acceptable pattern for *genuinely temporary, per-browser-session* state that a later phase will formalize — don't reach for it as a substitute for real schema when the state needs to persist across devices/logins.

## Frontend Approach

- Blade templates + Vite-bundled CSS/JS (Laravel 12 default), per the brief's stated stack.
- **Resolved 2026-07-02 — Vanilla JavaScript, no Alpine.js.** The Step 1 (Home page) brief explicitly named "Vanilla JavaScript" in the stack (alongside "no jQuery, no React, no Vue"), which supersedes this document's earlier Alpine.js recommendation. The mobile nav toggle (`resources/js/navbar.js`) is implemented in plain JS (`classList.toggle`, no framework). Apply the same approach to the Registration accordion when Phase 3 is built.
- **Resolved 2026-07-02 — no Tailwind, plain custom CSS.** The Step 1 brief explicitly said "No Bootstrap. No Tailwind components beyond what Laravel already ships." Rather than leave the default-scaffold Tailwind wiring in place unused, it was removed entirely: `@tailwindcss/vite` and `tailwindcss` uninstalled, `vite.config.js` no longer registers the Tailwind plugin, `resources/css/app.css` just imports our own token/base/component stylesheets. [DESIGN_SYSTEM.md](DESIGN_SYSTEM.md)'s tokens (colors, spacing, type scale) are implemented as plain CSS custom properties in `resources/css/variables.css`, consumed by hand-written component stylesheets under `resources/css/components/`. No CSS framework, no utility-class methodology.
- **CSS folder split, established Step 7:** `resources/css/components/` for anything reusable across pages (buttons, navbar, forms, data-display widgets like `status-badge`/`product-table`), vs. `resources/css/pages/` for layout specific to one page/module (currently just `vendor.css`) that wouldn't make sense anywhere else. When in doubt: if a second page would plausibly reuse the class, it's a component; if it's about how one module's specific page is arranged, it's page-specific.
- **File uploads use Laravel's `public` disk.** `php artisan storage:link` was run (Step 7) to symlink `public/storage` → `storage/app/public`. Product images store under `storage/app/public/products`, vendor logos under `storage/app/public/vendor-logos`. Models expose `Storage::disk('public')->url(...)` through accessors (`ProductImage::url`, `VendorProfile::logoUrl`) rather than calling the facade from Blade — see the Policies section above for the parallel "wrap it once" reasoning.

## Validation & Error Handling

- User input validated at the boundary (Form Requests), per this project's own "don't validate what can't happen" philosophy — no defensive checks deeper in the stack for things a Form Request already guaranteed.
- No custom error-handling abstractions beyond Laravel's defaults unless a specific need arises.

## Database Conventions (added 2026-07-02, Step 5)

- **DB transactions for multi-table writes:** anywhere a single user action needs to write to more than one table atomically (e.g. `RegisterController` creating a `users` row and, for vendors, a `vendor_profiles` row), wrap it in `DB::transaction()` — a partial write (user created, profile creation failed) is worse than the whole thing rolling back.
- **Unique slug generation:** when a user-supplied string needs a unique slug column (e.g. `vendor_profiles.store_slug` from `store_name`), generate the base via `Str::slug()`, then check-and-suffix (`-1`, `-2`, ...) until unique, rather than trusting the input is already unique. See `RegisterController::uniqueStoreSlug()` for the reference implementation.
- **Cascade deletes by default** on child records that have no meaning without their parent (`product_images`, `collection_items`, `wishlists`, `reviews`, `addresses` all `cascadeOnDelete()` their parent FK). `categories.parent_id` uses `nullOnDelete()` instead — deleting a parent category shouldn't cascade-delete its children, just orphan them to top-level.
- **Pivot-like tables with extra columns get a first-class model, not a bare `belongsToMany`:** `collection_items` has its own `display_order`, so it's modeled as `CollectionItem` (`hasMany`/`belongsTo`), not an implicit Eloquent pivot.
- **Naming collision to remember:** `App\Models\Collection` shares a name with Laravel's own `Illuminate\Support\Collection`. Fully qualify or alias imports in any file that needs both.
- **Row-locking for stock-sensitive writes, introduced 2026-07-03 (Step 10):** anywhere a write depends on a product's *current* stock (`CartService::addToCart()`/`updateQuantity()`, `InventoryService::deduct()`), re-fetch the product with `Product::query()->lockForUpdate()->findOrFail($id)` inside a `DB::transaction()`, and validate against that locked read — not the possibly-stale model instance already in memory. This serializes concurrent requests for the same product row (e.g. two buyers racing for the last unit) rather than letting both read the same stock figure and both succeed. Use this pattern for any future write that must never let two concurrent requests both pass a check that only one of them should.
- **Business-rule validation via `ValidationException`, introduced 2026-07-03 (Step 10), reused Step 11:** when a service-layer rule is violated (e.g. `CartService::assertStockAvailable()`, or Step 11's `CategoryService::archive()` guarding against a category that still has products/children), throw `Illuminate\Validation\ValidationException::withMessages([...])` rather than a bespoke exception class. Laravel's default exception handler already redirects back with the errors flashed to the session for any non-JSON request — the exact same mechanism a failed Form Request produces — so no controller-level try/catch is needed, and the error surfaces through the same `<x-flash-status>` component every other validation error already uses. Reach for a custom exception class only if a rule needs to be caught and handled differently from "show the user why this failed."
- **`SoftDeletes` used sparingly, established Step 11:** `Category` is the **only** model with `SoftDeletes` in this project (added specifically because `products.category_id` has a real `cascadeOnDelete()` risk). Don't reach for `SoftDeletes` as a default "safer delete" — the trait adds a global scope that changes query results for *every* existing query against that model, everywhere in the codebase, which is a much bigger blast radius than it looks at the call site that adds it. Add it only when there's a specific, articulable data-loss risk it fixes (as documented in DATABASE_BLUEPRINT.md's Administration section), and pair it with an application-level guard (e.g. `archive()` checking for dependent rows) rather than relying on `deleted_at` alone to make the delete safe.
- **Config-overlay pattern, introduced Step 11:** `SettingsService::applyOverlay()` (called once from `AppServiceProvider::boot()`, guarded by `Schema::hasTable('settings')` for pre-migration artisan runs) copies admin-saved values from the `settings` table onto specific `config()` paths at every request boot. This is how a Settings UI can let an administrator edit a value **without** any consuming code (`HybridRecommendationService`, `CheckoutService`) ever needing to know a database-backed settings system exists — they keep reading `config('recommendation.weights')`/`config('shipping.delivery_costs')` exactly as before. Use this pattern for any future admin-editable value that an existing service already reads via `config()`; don't have services read the `settings` table directly.

## Security Conventions (added 2026-07-02, Step 4)

- **Rate limiting on login:** 5 attempts per email+IP combination (`RateLimiter`, keyed via `LoginRequest::throttleKey()`), standard Laravel/Breeze pattern — brute-force protection is baseline hygiene for any real login form, not optional polish.
- **Generic failure messages:** a failed login shows "These credentials do not match our records" rather than specifying whether the email or password was wrong — avoids user enumeration.
- **Session regeneration:** `$request->session()->regenerate()` after login/registration, `regenerateToken()` + `invalidate()` on logout — prevents session fixation, standard Laravel practice.
- **Password hashing:** handled automatically via the `User` model's `'password' => 'hashed'` cast (Laravel 11+) — never call `Hash::make()` manually when creating/updating a `User` through Eloquent.
- **No `trans()`/localization helpers used for auth messages** — this project has no `lang/` directory (Laravel 12's slim skeleton doesn't ship one by default), and localization is explicitly out of scope for now (see UI_BLUEPRINT.md Future Improvements). Auth messages are plain literal strings; revisit if localization is ever prioritized.

## Custom Artisan Commands (added 2026-07-03, Step 9)

- `app/Console/Commands/` for one-off operational/analysis commands that don't belong behind a route — first one: `EvaluateRecommendations` (`recommendations:evaluate`). ## Logging Channels (added 2026-07-03, Step 9)

- Dedicated log channels (`config/logging.php`) for structured, high-volume, or operationally-distinct events that shouldn't mix into `laravel.log`'s general noise — first one: `recommendations` (daily rotation, 14-day retention), written to by `RecommendationService` on every recommendation-generation event (algorithm, user, result count, execution time, cache hit/miss). Use `Log::channel('name')->info(...)` rather than the default `Log::info(...)` when a feature's events are worth being able to `tail`/grep in isolation.

## Documentation Sync Rule

Every blueprint document in `docs/` is a **living document**. Whenever a decision changes — a new entity, a different navigation flow, a revised recommendation weight — the relevant `docs/*.md` file gets updated **in the same session as the decision**, plus an entry in [PROJECT_DECISIONS.md](PROJECT_DECISIONS.md) and/or [PROJECT_EVOLUTION.md](PROJECT_EVOLUTION.md) as appropriate. Docs are not allowed to drift from reality.
